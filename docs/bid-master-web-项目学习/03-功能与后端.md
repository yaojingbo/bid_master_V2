# 维度3：功能与后端代码对应表

## 功能：用户登录

作用：通过邮箱 + 密码登录，返回 accessToken 并下发 refresh cookie

触发条件：用户在 `/login` 提交登录表单

入口文件：src/backend/app/api/auth.py（路由 /auth + /login handler，约第 121 行）

处理逻辑：

1. API route 接收 LoginRequest（email、password），Pydantic 校验字段格式
2. 调用 pg_storage.get_user_by_email（infrastructure/pg_storage.py）查询 users 表
3. 校验密码哈希（bcrypt）
4. 调用 create_access_token + create_refresh_token（utils/jwt.py），签名用 JWT_SECRET
5. 通过 Next Route（app/api/auth/login/route.ts）写 httpOnly refresh_token cookie
6. 新增 audit_logs 行（action=login）
7. 返回 { access_token, user }

数据库表：

- users（路径：infrastructure/pg_storage.py 中的 CREATE TABLE），存邮箱、密码哈希、状态
- audit_logs（drizzle/0000_icy_maestro.sql），存登录行为审计

## 功能：注册 + 邮箱验证码

作用：新用户通过邮箱验证码注册

触发条件：用户在 `/register` 输入邮箱 → 发送验证码 → 输入验证码和密码 → 提交

入口文件：src/backend/app/api/auth.py（/send-code、/register handler）

处理逻辑：

1. /send-code 接收邮箱 → 生成 6 位数字码 → 存 redis/内存（60s TTL）→ 发邮件
2. /register 接收 email、code、password → 校验验证码 → bcrypt 哈希密码 → pg_storage.add_user 写 users 表
3. 颁发 access + refresh token（同登录）
4. 返回 { access_token, user }

数据库表：users

---

## 功能：文件上传（加密存储）

作用：上传招标 PDF/Word 文件，服务端加密落盘

触发条件：在 /extract、/simulate、/statistics 任一页面拖拽文件

入口文件：src/backend/app/api/files.py（/upload handler，约第 37 行）

处理逻辑：

1. API route 接收 multipart 文件，调用 FileService.validate_file（services/file_service.py）检查扩展名/大小
2. 读取文件字节 → EncryptionService.encrypt（services/encryption_service.py，Fernet）→ 写本地存储（StorageService 抽象，可切换 pg/mock）
3. pg_storage.add_document 插入 tender_documents 表（status=uploading → ready）
4. 返回 { id, filename, size, ... }

数据库表：tender_documents（drizzle/0000_icy_maestro.sql）— encrypted_path、encryption_key_id、status、metadata jsonb

## 功能：要素提取（AI 流式）

作用：从招标文件中提取关键要素（投标人资格、评标办法、合同条款等），SSE 流式返回

触发条件：/extract 页选文件 + 模板 + 要素清单 → 点击"开始提取"

入口文件：src/backend/app/api/extract.py（/element handler，约第 43 行）

处理逻辑：

1. API route 接收 ExtractRequest（file_id、template_id、elements[]），参数校验 + 权限检查（文件归属）
2. StreamingResponse 返回，SSE 事件流：progress / content / done / error
3. 调用 extract_service（services/extract_service.py）：
   a. FileService 读取 + decrypt
   b. extract_text_from_content 解析 PDF/Word 文本
   c. 文本过少 → ocr_service.is_vision_capable 走多模态 LLM OCR
   d. PromptBuilder.build_extract_system_prompt / build_extract_user_prompt（services/prompt_builder.py，模板来自 prompts/*.md）
   e. LiteLLMService 流式调用 LLM，按要素切片
   f. 解析 LLM JSON 输出：_parse_llm_json_response → _normalize_elements
4. 调用 add_extract 写入 analysis_results + extracted_elements 表
5. SSE 推送 done 事件（含 analysis_id）

数据库表：

- analysis_results（type=element_extract、status、result jsonb，FK→tender_documents）
- extracted_elements（name、content、confidence、position，FK→analysis_results，ON DELETE cascade）

## 功能：批量对比提取

作用：多份招标文件并排提取，对比要素差异

触发条件：/extract → 批量对比 Tab 选多文件 + 要素 → 提交

入口文件：src/backend/app/api/extract.py（/element/batch handler）

处理逻辑：

1. API route 接收 BatchExtractRequest（file_ids[]、elements[]）
2. 内部对每个 file_id 串行/并行调用 extract_service.extract_element（逻辑同上）
3. 汇总结果，按要素归并（相同要素列出各文件的内容差异）
4. 写 analysis_results（type=batch_extract）+ 多个 extracted_elements
5. SSE 推送 done

数据库表：analysis_results、extracted_elements

## 功能：门槛分析

作用：基于批量提取结果，识别"门槛性"要素（如必须满足的资质条件）

触发条件：/extract → 门槛分析 Tab → 选已完成的批量提取 → 提交

入口文件：src/backend/app/api/extract.py（/element/threshold handler）

处理逻辑：

1. API route 接收 ThresholdRequest（analysis_id、threshold_keywords[]）
2. 读取 analysis_results.result 缓存的批量结果
3. PromptBuilder.build_threshold_prompt 构造提示词
4. LiteLLMService 调用 LLM，识别并返回门槛要素列表
5. 写 analysis_results（type=threshold_analysis）

数据库表：analysis_results

## 功能：模拟编制（4 步流式）

作用：根据招标文件生成投标文件草稿，分 4 步流式输出

触发条件：/simulate → 选文件 → 创建任务 → 逐步点"执行"

入口文件：src/backend/app/api/simulate.py（/create + /{task_id}/step/{1..4} handler）

处理逻辑：

1. /create 接收 file_id → pg_storage.add_simulate 写 simulate_tasks 表 → 返回 { task_id }
2. step 1（PDF 转换）：FileService 读取 → 调用 PDF 转换服务（pypdf/pdf2docx）→ 写中间产物
3. step 2（要素提取）：直接调用 extract_service 复用提取能力（见上方）
4. step 3（对比分析）：PromptBuilder 构造对比 prompt → LiteLLMService 流式 LLM → 写 analysis_results
5. step 4（模拟编制）：接收 project_name、project_type、region 等参数 → PromptBuilder 构造编制 prompt → LiteLLMService 流式生成投标文件草稿 → Markdown 流式 SSE 推送 → 写 simulate_tasks.result
6. 每步返回 SSE 事件流

数据库表：simulate_tasks（位于 pg_storage.add_simulate/update_simulate），存步骤结果与最终产物

## 功能：开标 Excel 解析 + 统计

作用：解析开标 Excel，统计各投标人报价、排名、离散度等

触发条件：/statistics 上传开标记录 Excel

入口文件：src/backend/app/api/statistics.py（/analyze/upload handler，约第 625 行）

处理逻辑：

1. API route 接收 multipart Excel
2. StatisticsService.calculate_price_statistics（services/statistics_service.py）— 纯计算：
   a. openpyxl 解析 Excel（投标人、报价列）
   b. 排名、降序、均值、中位、CV（变异系数）、降价幅度
3. 写 analysis_results（type=opening_analysis）+ statistics（data jsonb）
4. 返回 { analysis_id, statistics }

数据库表：

- analysis_results
- statistics（FK→analysis_results，data jsonb 含排名/CV/降价等结构）

## 功能：AI 综合分析（开标）

作用：在 Excel 统计基础上，调用 LLM 生成综合分析文本（异常检测、投标策略建议）

触发条件：statistics 上传 Excel 后自动触发 /analyze/comprehensive/upload/start

入口文件：src/backend/app/api/statistics.py（/analyze/comprehensive/upload/start + /analysis-task/{task_id}）

处理逻辑：

1. /start 接收 analysis_id → 创建后台任务（TaskManager）→ 立即返回 task_id
2. 后台：PromptBuilder 构造综合分析 prompt（含统计结果 + 原始数据）
3. LiteLLMService 流式调用 LLM → 进度写入 task_store
4. 前端轮询 GET /analysis-task/{task_id} 拿到 ai_analysis 文本

数据库表：analysis_results（更新 result.ai_analysis 字段）、tasks（任务状态）

## 功能：AI 提供商设置 + 连接测试

作用：管理 AI provider/model 配置和 API Key，测试连接

触发条件：/settings 页切换 provider/model、保存 Key、点击"测试"

入口文件：src/backend/app/api/settings.py、src/backend/app/api/api_keys.py

处理逻辑：

1. GET /providers → 返回内置供应商列表（OpenAI、Anthropic、DeepSeek 等）+ 每个支持的模型
2. POST /providers/{name} → 保存 provider/model 默认配置到 ai_configurations 表
3. POST /api-keys → EncryptionService.encrypt 加密 Key → 存 ai_configurations 或独立 key 表
4. DELETE /api-keys/{provider} → 删除
5. POST /settings/test → 用当前 provider+model+key 调 LLMService.test_connection（services/llm_service.py）— 发一次最小请求，返回延迟/成功/失败
6. 写 ai_configurations.last_test_result

数据库表：ai_configurations

## 功能：数据库 CRUD（files/extracts/simulates/openings）

作用：统一管理 4 类数据，提供列表、详情、删除、预览、下载、批量下载

触发条件：/database 页切换 Tab、操作记录

入口文件：src/backend/app/api/database.py（/data/*）

处理逻辑：

1. GET /data/files → pg_storage.list_files → 读 tender_documents（分页、过滤、排序）
2. GET /data/extracts → 读 analysis_results WHERE type=element_extract
3. GET /data/simulates → 读 simulate_tasks
4. GET /data/openings → 读 analysis_results WHERE type=opening_analysis
5. DELETE /{resource}/{id} → pg_storage.delete_* → cascade 删除子表
6. GET /files/{id}/download → StorageService 读 + decrypt → 返回 FileResponse
7. GET /files/{id}/preview → 解析文本前 5KB → 返回 JSON
8. POST /files/batch-download → 多文件打包为 zip（jszip）→ 返回
9. GET /extracts/{id}/export-json → 读 analysis_results.result + extracted_elements → 返回 JSON

数据库表：tender_documents、analysis_results、extracted_elements、statistics、simulate_tasks（均通过 pg_storage 抽象）

## 功能：CLI 设备授权

作用：CLI 在本地发起授权，浏览器确认后获取 token

触发条件：用户运行 `bidmaster auth login` → CLI 调 start → 浏览器打开授权页

入口文件：src/backend/app/api/cli_auth.py（/cli-auth/*）

处理逻辑：

1. CLI 端（src/backend/app/cli.py）调 POST /cli-auth/start → 返回 device_code、expires_in
2. 用户在浏览器打开 /cli-auth?device_code=xxx → 点击"授权"
3. 前端 POST /cli-auth/authorize { device_code } → 服务端把 device_code 标记为已授权
4. CLI 端循环 POST /cli-auth/poll { device_code } → 拿到 token
5. CLI 把 token + refresh 存 ~/.bid-master/credentials.json（chmod 600）
6. 后续 CLI 调 API 时 Authorization: Bearer <token>

数据库表：cli_auth_codes（pg_storage.add_cli_code/mark_authorized）

## 功能：日志收集 + 展示

作用：收集前端 + 后端日志，统一展示，支持过滤清空

触发条件：服务运行中持续收集，/logs 页查看

入口文件：src/backend/app/api/logs.py（/logs）

处理逻辑：

1. 后端 middleware.log_requests（main.py）— 拦截所有 /api/* 请求（排除 /api/logs 自身）→ 写入 add_log
2. 前端 log-store.ts 收集 console + 自定义日志
3. 启动时 GET /api/logs 拉取历史（最多 200 条）
4. SSE 推流新日志（/api/logs/stream，可选）
5. DELETE /api/logs 清空

数据库表：log_entries（pg_storage.add_log，in-memory + DB 双写）
